# Assignments

# Main goal: use OOP approach to handle the visual objects. 

Each object should contain its own logic, and when we need to use it we call the object itself using one of its methods or property instead of having all the properties from all the objects in every place, and having big if\else\switch everywhere.

- Remove VisualBuilderElement, and instead have an interface per object, and combine interface when possible. For example, right now all the objects must have type, position, visible, alpha and panelId, so it appears in VisualBuilderElementBase. Similarly the Circle\Rectangle\Arrow all have the BasicShape interface with the extra width, height, color , and then finally, the arrow should have extra property for angle or orientation.

- The cell data should not contain a collection of properties. Instead it should have a refernce to the object itself. Then if we need some property we extract it from the object. If there is a property that many objects have (e.g. width\height) they should be under the same name \ method.

- Use the object reference from the cell data to do the rendering in the `GridCell.tsx` .
Rendering should depend on (1) data from the object and (2) extra style data (e.g. light\dark theme). Each object should provide its own rendering function. This can be either like:
`obj.render(extra_style_parameters)`
or have a central render registery where each object created needs to register its rendering function (e.g. Label and LabelView) and then call
`render_registry.render(obj_type, obj_data, extra_style_parameters)`
The important part is that GridCell.tsx should not know which object it works with, just that it has a render function. 

- Elements like VISUAL_ELEM_SCHEMA should be generated automatically. For example, after each component is defined (e.g. rect, circle, array etc), it registers itself and provide the data needed. Then, after they are all registered, the schema is generated.

- The `VisualBuilder.py` should be generated eventually automatically from the information in the VISUAL_ELEM_SCHEMA. For now start by having each class having its own serialization.

- The basic shapes python section should be easy to generate from VISUAL_ELEM_SCHEMA. Move them to a second autogenerated file.

- The same registration process should be done for the switch command inside the visualBuilderExecutor, namely create a dictionary from type name to the corresponding class.
